# The world needed yet another JavaScript library...

... and it has gotten yet another one in [scandiojs](https://github.com/scandio/scandiojs). The ubiquitous question arising seems to be: why would anybody possibly ever write another OpenSource JavaScript library in an age of GitHub, Node and raving sharing of code in general. Wouldn't it have been so much easier to pick [Underscore](http://underscorejs.org), [jQuery](http://jquery.com) and some small plugins which are flying around the inter-webs and merge them into one, to become a company's JavaScript library. In this case the one of [Scandio GmbH](www.scandio.de).

Lets a step back and reconsider ourselves. What is the difference between a library and a framework and how did most tiny libraries get to see the light of OpenSource? Mostly companies and their developers kept running into the same old problems during their every day development. Some might have used snippets, others might even have implemented the same thing a redundantly across various projects. This is surely not culpable but why should not everybody collaborate on one project [contributing](https://github.com/scandio/scandiojs/blob/master/CONTRIBUTING.md) his own solutions to others. Discuss issues and problems while trying to find a solution suiting everybody without anybody having to run into the same old booby traps once again. Sounds almost too saviour and good.

This is exactly what we tried to do: find out about common problems and workarounds people have been using over time and trying to incorporate them into one library: yet another JavaScript library. Nevertheless, this library is especially tailored to our needs while offering an encouraging guise for everybody to contribute back. It is intended to be modular and testable. Hopefully it will be used across the board and various project types.

Up to this stage it is build around 12 modules starting at conditional library and script loading, moving on to string manipulation and DOM-caching, allowing logging into the DOM while having standard utility functions and also easing timing and offering a [”php-bridge”](https://github.com/scandio/scandiojs/tree/master/php). Still, there is more such as a lightweight module definition for little reusable scripts and more little things.

It is all built using [Grunt](http://gruntjs.com/) and its ample set of available build-tasks, [Jasmine](http://pivotal.github.io/jasmine/) and [Testem](https://github.com/airportyh/testem) for testing and [TravisCI](https://travis-ci.org/) for continuous integration. Furthermore, all the works is hosted on [GitHub](https://github.com/scandio/scandiojs) waiting for anybody internally or even externally from Scandio to write a little extension. There is a mixin-function for it with [some examples](https://github.com/scandio/scandiojs/blob/master/test/featurerequests/src/template.js) already set up waiting.

Maybe yet another JavaScript library will end up being something useful for some people.
